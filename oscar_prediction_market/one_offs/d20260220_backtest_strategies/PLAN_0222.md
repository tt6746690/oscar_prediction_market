# Plan: Fix Calendar Bug + Redo Backtest Analysis

**Date:** 2025-02-22
**Status:** Planning

## Context

All 216 models (9 categories × 4 model types × 6 snapshots) trained for the
2025 backtest produce **identical predictions across all snapshots**. Root cause:
hardcoded `CAL = CALENDAR_2026` in `feature_engineering.py` line 442.

When `as_of_date` is in Jan–Feb 2025, all features with `available_from` dates
derived from CALENDAR_2026 (Jan–Mar 2026) are filtered out. Only static metadata
features survive (IMDb rating, RT score, budget, release month, etc.). Since
metadata doesn't change across snapshots, predictions are flat.

**Impact:** ALL current backtest results are based on models that never see
precursor signals. The "Directing LR +$368" finding is from a model that doesn't
know about DGA/PGA/SAG winners. All timing analysis is meaningless (model probs
don't vary). Every finding in the current README is suspect.

---

## 1. Fix the Calendar Bug

### 1.1 Bug Anatomy

There are **two distinct uses** of the global `CAL` in `feature_engineering.py`:

#### Use A: Import-time `available_from` dates (38 FeatureDefinition objects)

```python
CAL = CALENDAR_2026  # line 442

FeatureDefinition(
    name="oscar_total_nominations",
    available_from=CAL.oscar_nominations_date,  # = 2026-01-22
    ...
)
```

These dates are baked into `FeatureDefinition` objects at module import time.
`filter_features_by_availability(features, as_of_date)` compares `as_of_date`
against these dates. When `as_of_date=2025-01-23` and
`available_from=2026-01-22`, the feature is excluded.

**This is the primary bug.** It causes all time-varying features to be filtered
out when backtesting 2025.

#### Use B: Runtime `CAL.ceremony_number` in aggregate functions (4 references)

```python
def _precursor_wins_count(r, ctx):
    target_ceremony = CAL.ceremony_number  # = 98 (2026)
    if ctx.as_of_date is None or r.ceremony != target_ceremony:
        count += 1  # historical: count everything
    elif ctx.as_of_date >= CAL.precursor_winner_dates[award]:
        count += 1  # target ceremony: incremental
```

The "incremental" logic only activates for records from `target_ceremony`. For
2025 backtesting (ceremony 97), `r.ceremony != 98` is always `True`, so the
historical branch is taken. **This works correctly for 2025 by accident**, but
would break if we tried to do temporal backtesting for 2023 or 2024.

#### Secondary Bug: `as_of_date` not passed to `load_data()`

Both `train_predict.py` (line 744) and `evaluate_cv.py` (line 741) call
`load_data()` without `as_of_date`:

```python
df = load_data(model_type=model_type_for_features, raw_path=Path(args.raw_path))
```

This means `transform_dataset` gets `as_of_date=None`, and the aggregate
functions (`_precursor_wins_count`, `_precursor_nominations_count`) always take
the "historical" (non-incremental) path. Feature *selection* is filtered
separately via `filter_feature_set_by_availability()`, which correctly uses
`as_of_date` — but the feature *values* are not incremental.

For 2025 backtesting this has no effect (ceremony 97 always takes historical
path). For 2026 production via `build_model.py` it IS correctly passed (line
219). But the inconsistency is a latent bug.

### 1.2 Fix Approaches

#### Approach A: Semantic Availability Events (recommended)

Replace concrete `available_from: date` with a semantic event label. Resolve to
concrete dates at runtime given a calendar.

**Changes:**

```python
# New enum (or StrEnum) for availability events
class AvailabilityEvent(StrEnum):
    OSCAR_NOMINATIONS = "oscar_nominations"
    PRECURSOR_WINNER_EARLIEST = "precursor_winner_earliest"
    PRECURSOR_NOM_EARLIEST = "precursor_nom_earliest"
    # Per-award events for individual precursor features:
    PRECURSOR_WINNER_GG = "precursor_winner:GOLDEN_GLOBE"
    PRECURSOR_WINNER_DGA = "precursor_winner:DGA"
    PRECURSOR_NOM_PGA = "precursor_nom:PGA"
    # ... etc. (~20 values total)

# FeatureDefinition changes:
class FeatureDefinition(BaseModel):
    available_event: AvailabilityEvent | None = None  # replaces available_from: date

# New resolver:
def resolve_availability_date(
    event: AvailabilityEvent, calendar: AwardsCalendar
) -> date:
    ...

# filter functions gain a calendar parameter:
def filter_features_by_availability(
    features, as_of_date, calendar: AwardsCalendar
) -> list[FeatureDefinition]:
    ...

# TransformContext gains calendar:
class TransformContext(BaseModel):
    calendar: AwardsCalendar
    ...

# Runtime functions use ctx.calendar instead of global CAL
```

**Affected files:**
- `feature_engineering.py`: ~38 `available_from=` → `available_event=`, remove
  global `CAL`, add `AvailabilityEvent`, add `resolve_availability_date()`, update
  `TransformContext`, update `_precursor_wins_count`
  / `_precursor_nominations_count` to use `ctx.calendar`
- `data_loader.py`: `filter_feature_set_by_availability()` gains `calendar`
  param, `load_data()` and `prepare_features()` gain `calendar` param
- `train_predict.py`: pass `calendar` from CLI
- `evaluate_cv.py`: pass `calendar` from CLI
- `build_model.py`: pass `calendar` to subprocess commands
- `train_models.sh`: add `--ceremony-year 2025`

**Pros:**
- Correct by construction — no concrete dates baked in at import time
- Calendar becomes a required runtime parameter; impossible to forget
- `AvailabilityEvent` is self-documenting and greppable
- Works for any ceremony year without code changes
- Clean separation: features define *what* event, callers supply *when*

**Cons:**
- Largest refactor (~38 feature definitions + 6 files)
- All downstream callers need updating
- Need to define ~20 enum values for all distinct events
- `AvailabilityEvent` as StrEnum means parsing strings like
  `"precursor_winner:GOLDEN_GLOBE"` which is a bit hacky (or need ~20 explicit
  values)

#### Approach B: `set_target_calendar(calendar)` — Mutate Global State

Add a function that updates the global `CAL` and walks all `FeatureDefinition`
objects in `FEATURE_REGISTRY` to remap their `available_from` dates.

**Changes:**

```python
# Track mapping from feature name → calendar expression
_AVAILABILITY_MAP: dict[str, Callable[[AwardsCalendar], date]] = {}

def set_target_calendar(calendar: AwardsCalendar) -> None:
    """Reconfigure all feature availability dates for a new calendar."""
    global CAL
    CAL = calendar
    for name, expr_fn in _AVAILABILITY_MAP.items():
        FEATURE_REGISTRY[name].available_from = expr_fn(calendar)
```

Need to register each feature's availability expression when building lists.

**Pros:**
- Smallest diff — mostly additive, existing code structure preserved
- `FEATURE_REGISTRY`, `LR_FEATURES`, `GBT_FEATURES` etc. still work as-is
- Easy for callers: one `set_target_calendar(CALENDARS[2025])` call early

**Cons:**
- **Global mutable state** — not thread-safe, order-dependent
- Callers can forget to call it (same class of bug as current)
- Need to mutate Pydantic model fields (need `model_config =
  {"extra": "forbid", "frozen": False}` or use `object.__setattr__`)
- Building `_AVAILABILITY_MAP` requires a parallel data structure alongside
  the feature lists — can get out of sync
- Mutations to module-level lists (BASE_FEATURES etc.) are visible everywhere

#### Approach C: Factory Function for Feature Lists

Move all feature list construction into a function that takes a calendar:

```python
def build_all_features(cal: AwardsCalendar) -> dict[str, FeatureDefinition]:
    base = [
        FeatureDefinition(
            name="oscar_total_nominations",
            available_from=cal.oscar_nominations_date,
            ...
        ),
        ...
    ]
    ...
    return registry
```

**Pros:**
- Pure — no global mutable state
- Each invocation gets fresh, correctly-dated features
- Easy to test (pass different calendars, assert dates)
- Conceptually clean: features are a function of calendar

**Cons:**
- **Massive refactor** — 1200+ lines of feature definitions move inside a function
- All module-level exports (`BASE_FEATURES`, `LR_FEATURES`, `GBT_FEATURES`,
  `FEATURE_REGISTRY`) become either function-local or need a different API
- Every consumer of these exports must change
- Lambdas in feature transforms work but the function is enormous
- Loses the nice declarative structure of module-level lists

#### Approach D: Deferred Resolution via Property

Keep `available_from` as a field but make it computed from a stored event +
a module-level calendar reference:

```python
class FeatureDefinition(BaseModel):
    _available_event: str | None = None
    
    @property
    def available_from(self) -> date | None:
        if self._available_event is None:
            return None
        return _resolve(self._available_event, _current_calendar)
```

**Pros:**
- Transparent to consumers — `.available_from` still returns a date
- No need to change filter functions

**Cons:**
- Relies on mutable `_current_calendar` global (same as B)
- Pydantic computed fields + private attributes are awkward
- Mixes Pydantic model with runtime global state
- Hard to reason about

### 1.3 Recommendation

**Approach A (Semantic Availability Events)** is the right fix. It's more work
but eliminates the class of bug entirely. The `available_from: date` field
type-checks just fine but embeds a temporal assumption at import time that
silently breaks for other ceremony years. Making the calendar a required runtime
parameter means the code fails fast if you forget it.

Approach B is the pragmatic fallback if we want to move faster. It works but
leaves a fragile global mutation pattern.

**Decision needed:** A or B? I recommend A but it touches more files.

### 1.4 Additional Design Question: AvailabilityEvent representation

For Approach A, the ~38 `available_from=` usages map to roughly these distinct
calendar expressions:

| Expression | Count | Proposed Event |
|---|---|---|
| `CAL.oscar_nominations_date` | 24 | `OSCAR_NOMINATIONS` |
| `CAL.precursor_winner_dates[award]` | 7 | `PRECURSOR_WINNER_{award}` |
| `CAL.precursor_nomination_dates[award]` | 5 | `PRECURSOR_NOM_{award}` |
| `min(CAL.precursor_winner_dates.values())` | 2 | `PRECURSOR_WINNER_EARLIEST` |
| `min(CAL.precursor_nomination_dates.values())` | 2 | `PRECURSOR_NOM_EARLIEST` |

Two representation options:

1. **Flat StrEnum** with ~20 explicit values: `OSCAR_NOMINATIONS`,
   `PRECURSOR_WINNER_GOLDEN_GLOBE`, `PRECURSOR_WINNER_DGA`, etc.
   - Pro: explicit, IDE-discoverable, validates at definition time
   - Con: must add a new enum value whenever a new award is added

2. **Structured type**: `AvailabilityEvent = Literal["oscar_nominations"] |
   tuple[Literal["precursor_winner", "precursor_nom"], PrecursorAward]`
   - Pro: composable, doesn't need updating for new awards
   - Con: more complex type, harder to serialize

**Leaning toward option 1** (flat enum) since the set of awards is small and
stable.

## 4. Analysis Improvements

These are changes to `analyze.py`. Some are independent of the bug fix (code
quality), but the results only make sense after the fix.

### 4.1 Unified Config Analysis (new)

Currently `select_best_config()` picks one config by aggregate P&L. We need:

- **1 model + 1 config for all categories**: Not cherry-picked per-category bests.
  Show a table: for each category, the P&L of the unified best config. Some
  categories will lose money — that's fine and important to see.
- **Mean P&L**, not just aggregate: The current best config has `total_pnl` but
  we also need `mean_pnl_per_category` and `std` to understand variance.
- **Robustness**: Show how many "nearby" configs (e.g., same model, slightly
  different kelly_fraction) also make or lose money. If the profitable region is
  a tiny island in config space, that's worrying.

### 4.2 Brier vs P&L Scatter Fix

Current `plot_classification_vs_trading()`:
- Y-axis: **best P&L** (max across all trading configs) — cherry-picked
- Should be: **mean P&L** across configs (or P&L of the unified best config)

Also add:
- Kendall τ annotation (already present, good)
- Color by category, shape by model (or vice versa) for better readability

### 4.3 Timing Analysis Fix

Current `plot_timing_analysis()`:
- Only shows winner's `model_prob - market_prob` line
- Y-axis label is unclear ("Edge %")

Fix:
- Show **non-winners** too (in lighter color) — helps readers see whether the
  model correctly distinguishes winner from losers over time
- Y-axis should be "Model Probability (%)" not edge
- Overlay market probability for the winner as a separate line
- Add vertical lines for precursor event dates (GG winner, DGA winner, etc.)
  to show when new information arrives

### 4.4 Probability Trajectory Fix

Current `plot_probability_trajectories()`:
- All nominees on one subplot per model×category
- Solid/dashed lines for same nominee get different auto-cycle colors (bug)
- Hard to read with 10 nominees on one subplot

Fix:
- **One subplot per nominee** (or per model) — pick the cleaner layout
- Consistent colors: one color per nominee, solid=model, dashed=market
- Winner highlighted (thicker line, bright color)
- Add precursor event markers as before

### 4.5 Drop Nominee Correlation Plot

`plot_nominee_correlation()` computes Pearson r across ~6 snapshot dates. With
n=6, any correlation is meaningless. **Remove entirely.**

### 4.6 Trade-Level Deep Dive (new)

For the unified best config, show a **bet anatomy** table. For each trade:

| Category | Nominee | Date | Side | Price | Model Prob | Edge | Contracts | P&L | Winner? |
|---|---|---|---|---|---|---|---|---|---|

This requires saving `BacktestResult.trade_log` to CSV during `run_backtests.py`.
Currently trade logs exist in memory but are not persisted.

**Changes needed:**
- `run_backtests.py`: Save `trade_log` as CSV for each (cat, model, config)
- `analyze.py`: Load and display for the unified best config

### 4.7 Risk Analysis (new)

For deployment confidence, add:

- **Worst-case category**: How much do you lose on the worst single category?
- **Sharpe-like metric**: Mean P&L / std across categories
- **Trade count**: How many total trades? (fewer = more variance)
- **Max drawdown**: Within the backtest period, what's the worst cumulative P&L point?
- **Win rate**: % of trades that are profitable at settlement

### 4.8 Reorganize Summary Tables

Current `print_summary_table()` dumps a flat table. Restructure into a
narrative:

1. **Headline**: Best unified config → aggregate P&L → per-category breakdown
2. **Where we make money** and **where we lose money** — and why
3. **Model comparison**: Which model type is best? Is it consistent across
   categories?
4. **Config sensitivity**: How sensitive is P&L to config parameters?

### 4.9 Trade Inspection Utility (optional, lower priority)

A reusable utility in `trading/` that loads a backtest result directory and
produces a formatted trade report. Could be useful for both 2025 backtest
analysis and live 2026 monitoring.

Scope:
- `trading/inspect.py` or similar
- Takes a directory with prediction CSVs + market prices + trade log
- Produces a formatted report (Markdown or terminal output)
- Per-trade P&L breakdown, cumulative P&L curve, win rate, etc.

**Decision needed:** Is this worth building as a reusable utility, or should
the analysis just live in `analyze.py`?
